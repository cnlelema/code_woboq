173 /// Determine whether a call to an unprototyped functions under
174 /// the given calling convention should use the variadic
175 /// convention or the non-variadic convention.
176 ///
177 /// There's a good reason to make a platform's variadic calling
178 /// convention be different from its non-variadic calling
179 /// convention: the non-variadic arguments can be passed in
180 /// registers (better for performance), and the variadic arguments
181 /// can be passed on the stack (also better for performance). If
182 /// this is done, however, unprototyped functions *must* use the
183 /// non-variadic convention, because C99 states that a call
184 /// through an unprototyped function type must succeed if the
185 /// function was defined with a non-variadic prototype with
186 /// compatible parameters. Therefore, splitting the conventions
187 /// makes it impossible to call a variadic function through an
188 /// unprototyped type. Since function prototypes came out in the
189 /// late 1970s, this is probably an acceptable trade-off.
190 /// Nonetheless, not all platforms are willing to make it, and in
191 /// particularly x86-64 bends over backwards to make the
192 /// conventions compatible.
193 ///
194 /// The default is false. This is correct whenever:
195 /// - the conventions are exactly the same, because it does not
196 /// matter and the resulting IR will be somewhat prettier in
197 /// certain cases; or
198 /// - the conventions are substantively different in how they pass
199 /// arguments, because in this case using the variadic convention
200 /// will lead to C99 violations.
201 ///
202 /// However, some platforms make the conventions identical except
203 /// for passing additional out-of-band information to a variadic
204 /// function: for example, x86-64 passes the number of SSE
205 /// arguments in %al. On these platforms, it is desirable to
206 /// call unprototyped functions using the variadic convention so
207 /// that unprototyped calls to varargs functions still succeed.
208 ///
209 /// Relatedly, platforms which pass the fixed arguments to this:
210 /// A foo(B, C, D);
211 /// differently than they would pass them to this:
212 /// A foo(B, C, D, ...);
213 /// may need to adjust the debugger-support code in Sema to do the
214 /// right thing when calling a function with no know signature.
215 virtual bool isNoProtoCallVariadic(const CodeGen::CallArgList &args,
216 const FunctionNoProtoType *fnType) const;
