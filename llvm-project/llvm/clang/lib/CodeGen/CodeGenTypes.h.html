171 // The arrangement methods are split into three families:
172 // - those meant to drive the signature and prologue/epilogue
173 // of a function declaration or definition,
174 // - those meant for the computation of the LLVM type for an abstract
175 // appearance of a function, and
176 // - those meant for performing the IR-generation of a call.
177 // They differ mainly in how they deal with optional (i.e. variadic)
178 // arguments, as well as unprototyped functions.
179 //
180 // Key points:
181 // - The CGFunctionInfo for emitting a specific call site must include
182 // entries for the optional arguments.
183 // - The function type used at the call site must reflect the formal
184 // signature of the declaration being called, or else the call will
185 // go awry.
186 // - For the most part, unprototyped functions are called by casting to
187 // a formal signature inferred from the specific argument types used
188 // at the call-site. However, some targets (e.g. x86-64) screw with
189 // this for compatibility reasons.
190
191 const CGFunctionInfo &arrangeGlobalDeclaration(GlobalDecl GD);

261 /// "Arrange" the LLVM information for a call or type with the given
