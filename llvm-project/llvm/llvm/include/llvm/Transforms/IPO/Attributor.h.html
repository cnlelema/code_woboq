9// Attributor: An inter procedural (abstract) "attribute" deduction framework.
10//
11// The Attributor framework is an inter procedural abstract analysis (fixpoint
12// iteration analysis). The goal is to allow easy deduction of new attributes as
13// well as information exchange between abstract attributes in-flight.
14//
15// The Attributor class is the driver and the link between the various abstract
16// attributes. The Attributor will iterate until a fixpoint state is reached by
17// all abstract attributes in-flight, or until it will enforce a pessimistic fix
18// point because an iteration limit is reached.
19//
20// Abstract attributes, derived from the AbstractAttribute class, actually
21// describe properties of the code. They can correspond to actual LLVM-IR
22// attributes, or they can be more general, ultimately unrelated to LLVM-IR
23// attributes. The latter is useful when an abstract attributes provides
24// information to other abstract attributes in-flight but we might not want to
25// manifest the information. The Attributor allows to query in-flight abstract
26// attributes through the `Attributor::getAAFor` method (see the method
27// description for an example). If the method is used by an abstract attribute
28// P, and it results in an abstract attribute Q, the Attributor will
29// automatically capture a potential dependence from Q to P. This dependence
30// will cause P to be reevaluated whenever Q changes in the future.
31//
32// The Attributor will only reevaluated abstract attributes that might have
33// changed since the last iteration. That means that the Attribute will not
34// revisit all instructions/blocks/functions in the module but only query
35// an update from a subset of the abstract attributes.
36//
37// The update method `AbstractAttribute::updateImpl` is implemented by the
38// specific "abstract attribute" subclasses. The method is invoked whenever the
39// currently assumed state (see the AbstractState class) might not be valid
40// anymore. This can, for example, happen if the state was dependent on another
41// abstract attribute that changed. In every invocation, the update method has
42// to adjust the internal state of an abstract attribute to a point that is
43// justifiable by the underlying IR and the current state of abstract attributes
44// in-flight. Since the IR is given and assumed to be valid, the information
45// derived from it can be assumed to hold. However, information derived from
46// other abstract attributes is conditional on various things. If the justifying
47// state changed, the `updateImpl` has to revisit the situation and potentially
48// find another justification or limit the optimistic assumes made.
49//
50// Change is the key in this framework. Until a state of no-change, thus a
51// fixpoint, is reached, the Attributor will query the abstract attributes
52// in-flight to re-evaluate their state. If the (current) state is too
53// optimistic, hence it cannot be justified anymore through other abstract
54// attributes or the state of the IR, the state of the abstract attribute will
55// have to change. Generally, we assume abstract attribute state to be a finite
56// height lattice and the update function to be monotone. However, these
57// conditions are not enforced because the iteration limit will guarantee
58// termination. If an optimistic fixpoint is reached, or a pessimistic fix
59// point is enforced after a timeout, the abstract attributes are tasked to
60// manifest their result in the IR for passes to come.
61//
62// Attribute manifestation is not mandatory. If desired, there is support to
63// generate a single LLVM-IR attribute already in the AbstractAttribute base
64// class. In the simplest case, a subclass overloads
65// `AbstractAttribute::getManifestPosition()` and
66// `AbstractAttribute::getAttrKind()` to return the appropriate values. The
67// Attributor manifestation framework will then create and place a new attribute
68// if it is allowed to do so (based on the abstract state). Other use cases can
69// be achieved by overloading other abstract attribute methods.
70//
71//
72// The "mechanics" of adding a new "abstract attribute":
73// - Define a class (transitively) inheriting from AbstractAttribute and one
74// (which could be the same) that (transitively) inherits from AbstractState.
75// For the latter, consider the already available BooleanState and
76// IntegerState if they fit your needs, e.g., you require only a bit-encoding.
77// - Implement all pure methods. Also use overloading if the attribute is not
78// conforming with the "default" behavior: A (set of) LLVM-IR attribute(s) for
79// an argument, call site argument, function return value, or function. See
80// the class and method descriptions for more information on the two
81// "Abstract" classes and their respective methods.
82// - Register opportunities for the new abstract attribute in the
83// `Attributor::identifyDefaultAbstractAttributes` method if it should be
84// counted as a 'default' attribute.
85// - Add sufficient tests.
86// - Add a Statistics object for bookkeeping. If it is a simple (set of)
87// attribute(s) manifested through the Attributor manifestation framework, see
88// the bookkeeping function in Attributor.cpp.
89// - If instructions with a certain opcode are interesting to the attribute, add
90// that opcode to the switch in `Attributor::identifyAbstractAttributes`. This
91// will make it possible to query all those instructions through the
92// `InformationCache::getOpcodeInstMapForFunction` interface and eliminate the
93// need to traverse the IR repeatedly.
94//

124/// The fixpoint analysis framework that orchestrates the attribute deduction.
125///
126/// The Attributor provides a general abstract analysis framework (guided
127/// fixpoint iteration) as well as helper functions for the deduction of
128/// (LLVM-IR) attributes. However, also other code properties can be deduced,
129/// propagated, and ultimately manifested through the Attributor framework. This
130/// is particularly useful if these properties interact with attributes and a
131/// co-scheduled deduction allows to improve the solution. Even if not, thus if
132/// attributes/properties are completely isolated, they should use the
133/// Attributor framework to reduce the number of fixpoint iteration frameworks
134/// in the code base. Note that the Attributor design makes sure that isolated
135/// attributes are not impacted, in any way, by others derived at the same time
136/// if there is no cross-reasoning performed.
137///
138/// The public facing interface of the Attributor is kept simple and basically
139/// allows abstract attributes to one thing, query abstract attributes
140/// in-flight. There are two reasons to do this:
141/// a) The optimistic state of one abstract attribute can justify an
142/// optimistic state of another, allowing to framework to end up with an
143/// optimistic (=best possible) fixpoint instead of one based solely on
144/// information in the IR.
145/// b) This avoids reimplementing various kinds of lookups, e.g., to check
146/// for existing IR attributes, in favor of a single lookups interface
147/// provided by an abstract attribute subclass.
148///
149/// NOTE: The mechanics of adding a new "concrete" abstract attribute are
150/// described in the file comment.
151struct Attributor {
152 ~Attributor() { DeleteContainerPointers(AllAbstractAttributes); }
153
154 /// Run the analyses until a fixpoint is reached or enforced (timeout).
155 ///
156 /// The attributes registered with this Attributor can be used after as long
157 /// as the Attributor is not destroyed (it owns the attributes now).
158 ///
159 /// \Returns CHANGED if the IR was changed, otherwise UNCHANGED.
160 ChangeStatus run();
161
162 /// Lookup an abstract attribute of type \p AAType anchored at value \p V and
163 /// argument number \p ArgNo. If no attribute is found and \p V is a call base
164 /// instruction, the called function is tried as a value next. Thus, the
165 /// returned abstract attribute might be anchored at the callee of \p V.
166 ///
167 /// This method is the only (supported) way an abstract attribute can retrieve
168 /// information from another abstract attribute. As an example, take an
169 /// abstract attribute that determines the memory access behavior for a
170 /// argument (readnone, readonly, ...). It should use `getAAFor` to get the
171 /// most optimistic information for other abstract attributes in-flight, e.g.
172 /// the one reasoning about the "captured" state for the argument or the one
173 /// reasoning on the memory access behavior of the function as a whole.
174 template <typename AAType>
175 const AAType *getAAFor(AbstractAttribute &QueryingAA, const Value &V,
176 int ArgNo = -1) {
177 static_assert(std::is_base_of<AbstractAttribute, AAType>::value,
178 "Cannot query an attribute with a type not derived from "
179 "'AbstractAttribute'!");
180 assert(AAType::ID != Attribute::None &&
181 "Cannot lookup generic abstract attributes!");
182
183 // Determine the argument number automatically for llvm::Arguments.
184 if (auto *Arg = dyn_cast<Argument>(&V))
185 ArgNo = Arg->getArgNo();
186
187 // If a function was given together with an argument number, perform the
188 // lookup for the actual argument instead. Don't do it for variadic
189 // arguments.
190 if (ArgNo >= 0 && isa<Function>(&V) &&
191 cast<Function>(&V)->arg_size() > (size_t)ArgNo)
192 return getAAFor<AAType>(
193 QueryingAA, *(cast<Function>(&V)->arg_begin() + ArgNo), ArgNo);
194
195 // Lookup the abstract attribute of type AAType. If found, return it after
196 // registering a dependence of QueryingAA on the one returned attribute.
197 const auto &KindToAbstractAttributeMap = AAMap.lookup({&V, ArgNo});
198 if (AAType *AA = static_cast<AAType *>(
199 KindToAbstractAttributeMap.lookup(AAType::ID))) {
200 QueryMap[AA].insert(&QueryingAA);
201 return AA;
202 }
203
204 // If no abstract attribute was found and we look for a call site argument,
205 // defer to the actual argument instead.
206 ImmutableCallSite ICS(&V);
207 if (ICS && ICS.getCalledValue())
208 return getAAFor<AAType>(QueryingAA, *ICS.getCalledValue(), ArgNo);
209
210 // No matching attribute found
211 return nullptr;
212 }
213
214 /// Introduce a new abstract attribute into the fixpoint analysis.
215 ///
216 /// Note that ownership of the attribute is given to the Attributor. It will
217 /// invoke delete for the Attributor on destruction of the Attributor.
218 ///
219 /// Attributes are identified by
220 /// (1) their anchored value (see AA.getAnchoredValue()),
221 /// (2) their argument number (\p ArgNo, or Argument::getArgNo()), and
222 /// (3) their default attribute kind (see AAType::ID).
223 template <typename AAType> AAType &registerAA(AAType &AA, int ArgNo = -1) {
224 static_assert(std::is_base_of<AbstractAttribute, AAType>::value,
225 "Cannot register an attribute with a type not derived from "
226 "'AbstractAttribute'!");
227
228 // Determine the anchor value and the argument number which are used to
229 // lookup the attribute together with AAType::ID.
230 Value &AnchoredVal = AA.getAnchoredValue();
231 if (auto *Arg = dyn_cast<Argument>(&AnchoredVal))
232 ArgNo = Arg->getArgNo();
233
234 // Put the attribute in the lookup map structure and the container we use to
235 // keep track of all attributes.
236 AAMap[{&AnchoredVal, ArgNo}][AAType::ID] = &AA;
237 AllAbstractAttributes.push_back(&AA);
238 return AA;
239 }

241 /// Determine opportunities to derive 'default' attributes in \p F and create
242 /// abstract attribute objects for them.
243 ///
244 /// \param F The function that is checked for attribute opportunities.
245 /// \param InfoCache A cache for information queryable by the new attributes.
246 /// \param Whitelist If not null, a set limiting the attribute opportunities.
247 ///
248 /// Note that abstract attribute instances are generally created even if the
249 /// IR already contains the information they would deduce. The most important
250 /// reason for this is the single interface, the one of the abstract attribute
251 /// instance, which can be queried without the need to look at the IR in
252 /// various places.
253 void identifyDefaultAbstractAttributes(
254 Function &F, InformationCache &InfoCache,
255 DenseSet</* Attribute::AttrKind */ unsigned> *Whitelist = nullptr);
256
257private:
258 /// The set of all abstract attributes.
259 ///{
260 using AAVector = SmallVector<AbstractAttribute *, 64>;
261 AAVector AllAbstractAttributes;
262 ///}
263
264 /// A nested map to lookup abstract attributes based on the anchored value and
265 /// an argument positions (or -1) on the outer level, and attribute kinds
266 /// (Attribute::AttrKind) on the inner level.
267 ///{
268 using KindToAbstractAttributeMap = DenseMap<unsigned, AbstractAttribute *>;
269 DenseMap<std::pair<const Value *, int>, KindToAbstractAttributeMap> AAMap;
270 ///}
271
272 /// A map from abstract attributes to the ones that queried them through calls
273 /// to the getAAFor<...>(...) method.
274 ///{
275 using QueryMapTy =
276 DenseMap<AbstractAttribute *, SetVector<AbstractAttribute *>>;
277 QueryMapTy QueryMap;
278 ///}
279};

281/// Data structure to hold cached (LLVM-IR) information.
282///
283/// All attributes are given an InformationCache object at creation time to
284/// avoid inspection of the IR by all of them individually. This default
285/// InformationCache will hold information required by 'default' attributes,
286/// thus the ones deduced when Attributor::identifyDefaultAbstractAttributes(..)
287/// is called.
288///
289/// If custom abstract attributes, registered manually through
290/// Attributor::registerAA(...), need more information, especially if it is not
291/// reusable, it is advised to inherit from the InformationCache and cast the
292/// instance down in the abstract attributes.
293struct InformationCache {
294 /// A map type from opcodes to instructions with this opcode.
295 using OpcodeInstMapTy = DenseMap<unsigned, SmallVector<Instruction *, 32>>;
296
297 /// Return the map that relates "interesting" opcodes with all instructions
298 /// with that opcode in \p F.
299 OpcodeInstMapTy &getOpcodeInstMapForFunction(Function &F) {
300 return FuncInstOpcodeMap[&F];
301 }
302
303 /// A vector type to hold instructions.
304 using InstructionVectorTy = std::vector<Instruction *>;
305
306 /// Return the instructions in \p F that may read or write memory.
307 InstructionVectorTy &getReadOrWriteInstsForFunction(Function &F) {
308 return FuncRWInstsMap[&F];
309 }
310
311private:
312 /// A map type from functions to opcode to instruction maps.
313 using FuncInstOpcodeMapTy = DenseMap<Function *, OpcodeInstMapTy>;
314
315 /// A map type from functions to their read or write instructions.
316 using FuncRWInstsMapTy = DenseMap<Function *, InstructionVectorTy>;
317
318 /// A nested map that remembers all instructions in a function with a certain
319 /// instruction opcode (Instruction::getOpcode()).
320 FuncInstOpcodeMapTy FuncInstOpcodeMap;
321
322 /// A map from functions to their instructions that may read or write memory.
323 FuncRWInstsMapTy FuncRWInstsMap;
324
325 /// Give the Attributor access to the members so
326 /// Attributor::identifyDefaultAbstractAttributes(...) can initialize them.
327 friend struct Attributor;
328};

330/// An interface to query the internal state of an abstract attribute.
331///
332/// The abstract state is a minimal interface that allows the Attributor to
333/// communicate with the abstract attributes about their internal state without
334/// enforcing or exposing implementation details, e.g., the (existence of an)
335/// underlying lattice.
336///
337/// It is sufficient to be able to query if a state is (1) valid or invalid, (2)
338/// at a fixpoint, and to indicate to the state that (3) an optimistic fixpoint
339/// was reached or (4) a pessimistic fixpoint was enforced.
340///
341/// All methods need to be implemented by the subclass. For the common use case,
342/// a single boolean state or a bit-encoded state, the BooleanState and
343/// IntegerState classes are already provided. An abstract attribute can inherit
344/// from them to get the abstract state interface and additional methods to
345/// directly modify the state based if needed. See the class comments for help.
346struct AbstractState {
347 virtual ~AbstractState() {}
348
349 /// Return if this abstract state is in a valid state. If false, no
350 /// information provided should be used.
351 virtual bool isValidState() const = 0;
352
353 /// Return if this abstract state is fixed, thus does not need to be updated
354 /// if information changes as it cannot change itself.
355 virtual bool isAtFixpoint() const = 0;
356
357 /// Indicate that the abstract state should converge to the optimistic state.
358 ///
359 /// This will usually make the optimistically assumed state the known to be
360 /// true state.
361 virtual void indicateOptimisticFixpoint() = 0;
362
363 /// Indicate that the abstract state should converge to the pessimistic state.
364 ///
365 /// This will usually revert the optimistically assumed state to the known to
366 /// be true state.
367 virtual void indicatePessimisticFixpoint() = 0;
368};

370/// Base struct for all "concrete attribute" deductions.
371///
372/// The abstract attribute is a minimal interface that allows the Attributor to
373/// orchestrate the abstract/fixpoint analysis. The design allows to hide away
374/// implementation choices made for the subclasses but also to structure their
375/// implementation and simplify the use of other abstract attributes in-flight.
376///
377/// To allow easy creation of new attributes, most methods have default
378/// implementations. The ones that do not are generally straight forward, except
379/// `AbstractAttribute::updateImpl` which is the location of most reasoning
380/// associated with the abstract attribute. The update is invoked by the
381/// Attributor in case the situation used to justify the current optimistic
382/// state might have changed. The Attributor determines this automatically
383/// by monitoring the `Attributor::getAAFor` calls made by abstract attributes.
384///
385/// The `updateImpl` method should inspect the IR and other abstract attributes
386/// in-flight to justify the best possible (=optimistic) state. The actual
387/// implementation is, similar to the underlying abstract state encoding, not
388/// exposed. In the most common case, the `updateImpl` will go through a list of
389/// reasons why its optimistic state is valid given the current information. If
390/// any combination of them holds and is sufficient to justify the current
391/// optimistic state, the method shall return UNCHAGED. If not, the optimistic
392/// state is adjusted to the situation and the method shall return CHANGED.
393///
394/// If the manifestation of the "concrete attribute" deduced by the subclass
395/// differs from the "default" behavior, which is a (set of) LLVM-IR
396/// attribute(s) for an argument, call site argument, function return value, or
397/// function, the `AbstractAttribute::manifest` method should be overloaded.
398///
399/// NOTE: If the state obtained via getState() is INVALID, thus if
400/// AbstractAttribute::getState().isValidState() returns false, no
401/// information provided by the methods of this class should be used.
402/// NOTE: The Attributor currently runs as a call graph SCC pass. Partially to
403/// this *current* choice there are certain limitations to what we can do.
404/// As a general rule of thumb, "concrete" abstract attributes should *for
405/// now* only perform "backward" information propagation. That means
406/// optimistic information obtained through abstract attributes should
407/// only be used at positions that precede the origin of the information
408/// with regards to the program flow. More practically, information can
409/// *now* be propagated from instructions to their enclosing function, but
410/// *not* from call sites to the called function. The mechanisms to allow
411/// both directions will be added in the future.
412/// NOTE: The mechanics of adding a new "concrete" abstract attribute are
413/// described in the file comment.
414struct AbstractAttribute {
415
416 /// The positions attributes can be manifested in.
417 enum ManifestPosition {
418 MP_ARGUMENT, ///< An attribute for a function argument.
419 MP_CALL_SITE_ARGUMENT, ///< An attribute for a call site argument.
420 MP_FUNCTION, ///< An attribute for a function as a whole.
421 MP_RETURNED, ///< An attribute for the function return value.
422 };
423
424 /// An abstract attribute associated with \p AssociatedVal and anchored at
425 /// \p AnchoredVal.
426 ///
427 /// \param AssociatedVal The value this abstract attribute is associated with.
428 /// \param AnchoredVal The value this abstract attributes is anchored at.
429 /// \param InfoCache Cached information accessible to the abstract attribute.
430 AbstractAttribute(Value *AssociatedVal, Value &AnchoredVal,
431 InformationCache &InfoCache)
432 : AssociatedVal(AssociatedVal), AnchoredVal(AnchoredVal),
433 InfoCache(InfoCache) {}
434
435 /// An abstract attribute associated with and anchored at \p V.
436 AbstractAttribute(Value &V, InformationCache &InfoCache)
437 : AbstractAttribute(&V, V, InfoCache) {}
438
439 /// Virtual destructor.
440 virtual ~AbstractAttribute() {}
441
442 /// Initialize the state with the information in the Attributor \p A.
443 ///
444 /// This function is called by the Attributor once all abstract attributes
445 /// have been identified. It can and shall be used for task like:
446 /// - identify existing knowledge in the IR and use it for the "known state"
447 /// - perform any work that is not going to change over time, e.g., determine
448 /// a subset of the IR, or attributes in-flight, that have to be looked at
449 /// in the `updateImpl` method.
450 virtual void initialize(Attributor &A) {}
451
452 /// Return the internal abstract state for inspection.
453 virtual const AbstractState &getState() const = 0;
454
455 /// Return the value this abstract attribute is anchored with.
456 ///
457 /// The anchored value might not be the associated value if the latter is not
458 /// sufficient to determine where arguments will be manifested. This is mostly
459 /// the case for call site arguments as the value is not sufficient to
460 /// pinpoint them. Instead, we can use the call site as an anchor.
461 ///
462 ///{
463 Value &getAnchoredValue() { return AnchoredVal; }
464 const Value &getAnchoredValue() const { return AnchoredVal; }
465 ///}
466
467 /// Return the llvm::Function surrounding the anchored value.
468 ///
469 ///{
470 Function &getAnchorScope();
471 const Function &getAnchorScope() const;
472 ///}
473
474 /// Return the value this abstract attribute is associated with.
475 ///
476 /// The abstract state usually represents this value.
477 ///
478 ///{
479 virtual Value *getAssociatedValue() { return AssociatedVal; }
480 virtual const Value *getAssociatedValue() const { return AssociatedVal; }
481 ///}
482
483 /// Return the position this abstract state is manifested in.
484 virtual ManifestPosition getManifestPosition() const = 0;
485
486 /// Return the kind that identifies the abstract attribute implementation.
487 virtual Attribute::AttrKind getAttrKind() const = 0;
488
489 /// Return the deduced attributes in \p Attrs.
490 virtual void getDeducedAttributes(SmallVectorImpl<Attribute> &Attrs) const {
491 LLVMContext &Ctx = AnchoredVal.getContext();
492 Attrs.emplace_back(Attribute::get(Ctx, getAttrKind()));
493 }
494
495 /// Helper functions, for debug purposes only.
496 ///{
497 virtual void print(raw_ostream &OS) const;
498 void dump() const { print(dbgs()); }
499
500 /// This function should return the "summarized" assumed state as string.
501 virtual const std::string getAsStr() const = 0;
502 ///}
503
504 /// Allow the Attributor access to the protected methods.
505 friend struct Attributor;
506
507protected:
508 /// Hook for the Attributor to trigger an update of the internal state.
509 ///
510 /// If this attribute is already fixed, this method will return UNCHANGED,
511 /// otherwise it delegates to `AbstractAttribute::updateImpl`.
512 ///
513 /// \Return CHANGED if the internal state changed, otherwise UNCHANGED.
514 ChangeStatus update(Attributor &A);
515
516 /// Hook for the Attributor to trigger the manifestation of the information
517 /// represented by the abstract attribute in the LLVM-IR.
518 ///
519 /// \Return CHANGED if the IR was altered, otherwise UNCHANGED.
520 virtual ChangeStatus manifest(Attributor &A);
521
522 /// Return the internal abstract state for careful modification.
523 virtual AbstractState &getState() = 0;
524
525 /// The actual update/transfer function which has to be implemented by the
526 /// derived classes.
527 ///
528 /// If it is called, the environment has changed and we have to determine if
529 /// the current information is still valid or adjust it otherwise.
530 ///
531 /// \Return CHANGED if the internal state changed, otherwise UNCHANGED.
532 virtual ChangeStatus updateImpl(Attributor &A) = 0;
533
534 /// The value this abstract attribute is associated with.
535 Value *AssociatedVal;
536
537 /// The value this abstract attribute is anchored at.
538 Value &AnchoredVal;
539
540 /// The information cache accessible to this abstract attribute.
541 InformationCache &InfoCache;
542};
